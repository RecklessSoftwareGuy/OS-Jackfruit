## 🏗️ 1. Theoretical Foundations

### 📦 A. Container Isolation (Namespaces)
Linux containers are NOT virtual machines. They are regular processes isolated by **Linux Namespaces**.
- **PID Namespace**: Isolates process IDs. The container sees itself as PID 1, but the host sees it as a high-numbered PID.
- **UTS Namespace**: Isolates hostname and domain name.
- **Mount Namespace**: Isolates mount points. Combined with `chroot` or `pivot_root`, it provides **Filesystem Isolation**.
- **Difference from VMs**: Containers share the host kernel; VMs run a guest kernel on hypervisor-emulated hardware.

### 🧵 B. Concurrent Logging & Synchronization
- **Bounded Buffer**: A shared queue with a fixed capacity. It prevents "buffer bloat" and backpressures producers if consumers are slow.
- **Mutex (Mutual Exclusion)**: Ensures only one thread accesses the shared buffer at a time.
- **Condition Variables (`pthread_cond_t`)**: Used for "signaling". 
    - `cond_empty`: Producer waits if the buffer is full.
    - `cond_fill`: Consumer waits if the buffer is empty.
- **Race Condition**: If two threads modify `buffer_count` simultaneously, the value becomes corrupt. Mutexes prevent this.

### 🧠 C. Memory Monitoring (RSS vs Virtual)
- **RSS (Resident Set Size)**: The actual physical memory (RAM) a process is using. Our kernel monitor tracks this.
- **Virtual Memory**: The total address space a process *thinks* it has. It includes swapped-out pages and shared libraries.
- **Policy Enforcement**:
    - **Soft Limit**: A threshold for warnings. The process is allowed to continue but "notified."
    - **Hard Limit**: A strict cutoff. Exceeding this triggers immediate termination (`SIGKILL`) to protect the host.

### ⚖️ D. Scheduling (CFS)
The **Completely Fair Scheduler (CFS)** is the default Linux scheduler.
- **vruntime (Virtual Runtime)**: CFS tracks how much time a process has spent on the CPU. It always picks the process with the *lowest* vruntime to run next.
- **Nice Values (-20 to 19)**:
    - `nice=0` (default) has a weight of **1024**.
    - `nice=10` has a weight of **110**.
    - CFS assigns CPU time proportionally to these weights.
- **Sleeper Fairness**: I/O-bound processes sleep often; their vruntime stays low. When they wake, they get scheduled immediately, making them feel "responsive."

## 🛠️ 2. Implementation Walkthrough

### 🚀 The Supervisor (`engine.c`)
1. **Creation**: Uses `clone(CLONE_NEWPID | CLONE_NEWNS | ...)` to create a new process in its own namespaces.
2. **Setup**: The child calls `chroot()` to its rootfs, `mount("proc"...)` to see its own process list, and then `execvp()` to run the command.
3. **Reaping**: The supervisor uses `waitpid()` or a `SIGCHLD` handler to clean up exited containers and avoid **zombie processes**.

### 🛡️ The Kernel Monitor (`monitor.c`)
1. **Registration**: User-space sends the container's Host PID via `ioctl(fd, REGISTER_PID, ... )`.
2. **Linked List**: The kernel stores tracked PIDs in a thread-safe linked list.
3. **Enforcement**: A background kernel thread or timer periodically checks the RSS of every process in the list. If `RSS > hard_limit`, it sends a `SIGKILL` using `send_sig()`.

## ❓ 3. Potential Examiner Questions (Q&A)

### 🔹 Level 1: Basics
**Q: Why use `clone()` instead of `fork()` for containers?**
**A:** `fork()` only creates a copy of the process. `clone()` allows us to specify flags like `CLONE_NEWPID` and `CLONE_NEWNET` to create new namespaces, which are essential for container isolation.

**Q: What is a zombie process, and how does your project avoid them?**
**A:** A zombie is a terminated process whose entry remains in the process table so the parent can read its exit status. Our supervisor avoids them by "reaping" them—calling `wait()` or `waitpid()` when a child exits.

### 🔸 Level 2: Intermediate
**Q: How does the container see its own PID as 1?**
**A:** This is the effect of the **PID Namespace**. Every PID namespace has its own numbering. The process is PID 1 inside the namespace but has a different, unique PID (e.g., 5432) on the host.

**Q: Why implement memory limits in the Kernel (LKM) instead of User-space?**
**A:** Security and Reliability. A user-space monitor might be delayed or killed. The kernel has direct access to process accounting structures (`task_struct`, `mm_struct`) and can enforce limits with 100% authority and minimal latency.

**Q: In your Bounded Buffer, what happens if the Consumer is slower than the Producer?**
**A:** The buffer will eventually fill up. Because it is "bounded," the Producer thread will block on a condition variable (`pthread_cond_wait`) until the Consumer removes an item and signals that there is space.

### 🔥 Level 3: Expert / Advanced
**Q: Does `chroot` provide perfect isolation?**
**A:** No. `chroot` only changes the root directory. A process with root privileges can potentially "break out." Modern runtimes use `pivot_root` (which we mentioned in the guide) and Mount Namespaces to provide a more secure filesystem jail.

**Q: Why did your I/O-bound process finish 'before' the CPU-bound one in Task 5?**
**A:** It finished sooner in wall-clock time because it wasn't waiting for the CPU; it was waiting for I/O and then waking up immediately due to the **CFS Sleeper Bonus**. It doesn't mean it's "faster" at computing; it just means it efficiently uses the gaps in CPU demand.

**Q: How does `ioctl` work as an IPC mechanism?**
**A:** System calls like `read`/`write` are for data streams. `ioctl` (Input/Output Control) is for "out-of-band" commands. It allows user-space to send structured requests (like "Register this PID with these limits") directly to a device driver or kernel module.
